# Scenario

You are the senior .NET backend developer on a cloud integration project. Two enterprise SaaS
platforms need to exchange data in near real-time:

| **SYSTEM** | **ROLE** |
| -------- | -------- |
| FieldOps | Source platform. Publishes equipment events to Azure Service Bus when assets are registered on site or checked in/out by operators. |
| AssetHub | Target platform. Receives equipment records via a REST API secured with OAuth 2.0. Equipment is managed at project level. All writes are scoped to a specific project. |

Your job is to build the integration middleware hosted on Azure that:

- **Subscribes** to the Azure Service Bus and receives equipment events from FieldOps in near real-time
- **Transforms** the event payload into an AssetHub-compatible equipment record
- **Authenticates** with AssetHub via OAuth 2.0 client credentials and manages the token lifecycle
- **Writes** equipment records to AssetHub via a REST API
- **Handles failures** using Polly retry and circuit breaker policies with dead-letter queue support
- **Provisions** all required Azure resources using Bicep infrastructure-as-code

> ### **IMPORTANT**
> Do not connect to real external services. Mock all dependencies. Use the Azure Service Bus Emulator or
> WireMock.Net locally. All external dependencies must be injectable and replaceable.

## Domain rules — read carefully
The following rules apply across all tasks and will be checked in the code review:
- **Asset ID** is generated by the middleware — not supplied by FieldOps. Format: `Make-Model-SerialNumber` e.g. `Caterpillar-320-SN-9901`. Hyphen separator. Preserve original casing.
- **Ownership must** always be hardcoded to "Subcontracted" — never mapped from the event payload. This is a fixed business rule enforced by the target system.
- **Dedup check**  must always run before any create. Search AssetHub by Asset ID. If a match is found — log a validation error and stop. If no match — proceed with create.
- `projectId`  is included in every FieldOps event. It must be used as the `{projectId}` path parameter in all AssetHub project-scoped API calls.
- **Token expiry**  AssetHub tokens expire after 5400 seconds (90 minutes). Refresh proactively at 5000 seconds — not reactively on 401.

## Event payload reference

The following JSON structures represent the two event types published by FieldOps to the Azure Service Bus:

### `asset.registration.submitted`
```json
{
  "eventType":  "asset.registration.submitted",
  "eventId":    "evt-a1b2c3",
  "projectId":  "proj-9001",
  "siteRef":    "SITE-AU-042",
  "fields": {
    "assetName":    "Caterpillar 320 Excavator",
    "make":         "Caterpillar",
    "model":        "320",
    "serialNumber": "SN-9901",
    "yearMfg":      "2021",
    "category":     "Earthmoving",
    "type":         "Excavator",
    "ratePerHour":  "220.00",
    "supplier":     "Hastings Deering Pty Ltd"
  },
  "imageUrl": "https://storage.example.com/assets/SN-9901.jpg"
}
```

### `asset.checkin.updated`
```json
{
  "eventType":    "asset.checkin.updated",
  "eventId":      "evt-d4e5f6",
  "projectId":    "proj-9001",
  "siteRef":      "SITE-AU-042",
  "serialNumber": "SN-9901",
  "make":         "Caterpillar",
  "model":        "320",
  "checkInDate":  "2026-05-06T07:00:00+10:00",
  "checkOutDate": null
}
```

> ### ONSITE DERIVATION RULE
> `checkInDate` present + `checkOutDate` is `null` → onsite = `true` (asset currently on site). `checkOutDate` present (non-null) → onsite = `false` (asset has checked out). checkInDate is null → throw ValidationException — invalid event.

## AssetHub REST API reference

The AssetHub API uses **OAuth 2.0 client credentials.**
All calls require `Authorization: Bearer {token}` and `X-Company-Id: {companyId}` headers. 
Base URL: `https://api.assethub.example.com`.

| ENDPOINT | METHOD(S) | PURPOSE |
| -------- | --------- | ------- |
| `/oauth/token` | POST | Client credentials token fetch |
| `/v1/companies/{companyId}/asset-statuses` | GET | Fetch active status id — cache on startup |
| `/v1/projects/{projectId}/assets?search={assetId}` | GET | Dedup check — exact match on assetId required |
| `/v1/projects/{projectId}/assets` | POST | Create new asset record |
| `/v1/projects/{projectId}/assets/{id}` | PATCH | Update fields or onsite status |
| `/v1/projects/{projectId}/assets/{id}/attachments` | POST | Upload asset profile photo |

# Tasks
Complete all four tasks within a single .NET solution. Partial credit is awarded. Bonus requirements are
optional and scored only if the core requirements are complete.

## Task 1 - Event Subscriber - Azure Service Bus
Build a hosted service that subscribes to the Azure Service Bus and processes incoming equipment
events. The subscriber must handle both event types, route them correctly, and deal with errors without
taking down the host process.

### **Requirements:**
- Subscribe to the service bus and process events continuously in the background
- Route each event type to the correct downstream handler
- Handle processing failures gracefully - failed messages should be retried by the bus, not silently dropped
- Log each event received along with its outcome
- All connection details must come from configuration - nothing hardcoded

### **Optional:**
- Make the concurrency limit configurable without a code change
- Carry the event ID through the log context so all log entries for a single event can be correlated

## Task 2 - AssetHub API Client - OAuth 2.0 + Asset Endpoints
Build a typed HTTP client for the AssetHub REST API. The client must authenticate via OAuth 2.0 client
credentials, keep the token fresh without manual intervention, and implement the asset endpoints. It
must be cleanly injectable and testable.

### **Requirements:**
- Authenticate using OAuth 2.0 client credentials. Store the token and refresh it before it expires — do not wait for a 401 to trigger a refresh. The token response includes the fields you need to calculate when it expires (see API response schemas section).
- On a 401 response - recover automatically by fetching a new token and retrying the failed request once
- Attach the required auth and company headers to every request
- Before creating an asset - search by Asset ID to prevent duplicates. An empty result means safe to create
- Implement create, update, and photo upload for asset records
- The client must be injectable and replaceable with a test double

### **Optional:**
- Handle rate limiting responses gracefully rather than failing immediately

## Task 3 - Data Transformer - Field Mapping and Validation
Build a component that takes a raw inbound event and produces a clean, validated payload ready to be
sent to the target system. The component should be pure transformation logic with no external
dependencies.

### **Requirements:**
- Map fields from the source event to the target payload structure
- Validate mandatory fields — fail clearly and identify what is missing
- Derive or generate any fields that are not directly present in the source event
- Normalise string values consistently
- Handle optional or nullable fields defensively — a missing value should degrade gracefully, not crash
- Apply any fixed business rules that must always hold regardless of what the source event contains

### **Optional:**
- Collect all validation errors before returning rather than failing on the first one
- Write unit tests for this component

## Task 4 - Resilience - Retry, Circuit Breaker + DLQ Processor
Make the outbound API calls resilient to transient failures. Build a dead-letter queue processor so failed
events can be replayed without manual intervention. All resilience behaviour must be configurable
without a code change.

### **Requirements:**
- Wrap outbound API calls with a retry policy — use exponential backoff with jitter. Only retry on transient errors (server errors, timeouts, rate limiting). Do not retry on client errors.
- Add a circuit breaker - stop hammering a failing service. Log clearly when the circuit opens.
- Build a processor that reads messages from the dead-letter queue and re-queues them for retry
- Re-processing must be safe to run multiple times - replaying an event that already succeeded must not create a duplicate
- Expose an endpoint to trigger the DLQ replay manually
- All retry and circuit breaker thresholds must be tunable via configuration

### **Optional:**
•  Use the .NET 10 built-in resilience pipeline (Microsoft.Extensions.Http.Resilience) rather than adding a third-party library
•  Expose a status endpoint showing last sync time, counts for today, and current circuit state

# What we look for at senior level

| AREA | WHAT GOOD LOOKS LIKE AT SENIOR LEVEL |
| ---- | ------------------------------------ |
| Architecture | Domain logic separated from infrastructure. Application layer defines interfaces. Infrastructure layer implements them. No business logic in controllers or hosted services. |
| Dependency injection | Everything in the DI container. No `new()` on services. `IOptions<T>` for all configuration binding. `IHttpClientFactory` for HTTP clients — never `new HttpClient()`. |
| Async correctness | `async`/`await` used correctly throughout the call chain. No `.Result` or `.Wait()`. `CancellationToken` passed through. `ConfigureAwait(false)` in library code. | 
| Error handling | Specific exception types used — not `catch(Exception)`. Domain exceptions separate from infrastructure exceptions. Structured logging with correlation IDs. No swallowed exceptions. |
| Security | No secrets in code or appsettings. Managed Identity used — not connection strings. Key Vault references in App Service configuration. Nothing sensitive committed to the repository. |
| README | Clear local setup and run instructions. Assumptions documented. Architecture decisions explained. Honest about what was not completed and why. |

## Local development guidance
The AssetHub API does not exist as a real service. Use WireMock.Net as your primary mock. A `docker-compose.yml` that starts the mock server is welcomed but not required.

| APPROACH | HOW IT WORKS | RECOMMENDED FOR |
| -------- | ------------ | --------------- |
| WireMock.Net | Start a local HTTP mock server. Configure route handlers for each endpoint. Point `BaseUrl` config to `localhost:9090`. | Primary integration — tests real HTTP, OAuth, and response deserialisation |

# Minimal WireMock setup example:

```csharp
// Install: dotnet add package WireMock.Net  (supports .NET 10)

var server = WireMockServer.Start(9090);

// Token endpoint
server.Given(Request.Create().WithPath("/oauth/token").UsingPost())
  .RespondWith(Response.Create().WithStatusCode(200).WithBodyAsJson(new {
    access_token = "mock-token",
    token_type   = "Bearer",
    expires_in   = 5400,
    created_at   = DateTimeOffset.UtcNow.ToUnixTimeSeconds()
  }));

// Category lookup — returns one existing category
server.Given(Request.Create()
  .RespondWith(Response.Create().WithStatusCode(200).WithBodyAsJson(new {
    data = new[] { new { id = 1, name = "Earthmoving" } }
  }));

// Asset search — empty = no match = safe to create
server.Given(Request.Create()
  .WithPath("/v1/projects/*/assets").UsingGet())
  .RespondWith(Response.Create().WithStatusCode(200).WithBodyAsJson(new {
    data = Array.Empty<object>()
  }));

// Asset create
server.Given(Request.Create()
  .WithPath("/v1/projects/*/assets").UsingPost())
  .RespondWith(Response.Create().WithStatusCode(201).WithBodyAsJson(new {
    data = new { id = "asset-001", assetId = "Caterpillar-320-SN-9901" }
  }));

// appsettings.Development.json
// "AssetHub": { "BaseUrl": "http://localhost:9090" }
```

- **Azure Service Bus** — use the official Microsoft Azure Service Bus Emulator Docker image, or mock `ServiceBusProcessor` in your hosted service. A `docker-compose.yml` with the emulator is welcomed.
- **AssetHub API** — the AssetHub API does not exist as a real service. You must mock it locally. The recommended approach is **`WireMock.Net`** — start a local mock server on port 9090 that responds to the endpoints listed in the API reference table, and set `BaseUrl = http://localhost:9090` in `appsettings.Development.json`.
- **Key Vault** — use `appsettings.Development.json` for local secrets. Document your local vs deployed config approach in the **`README.md`**.
- **Secrets** — never commit real credentials. Use placeholder values. A `.gitignore` that excludes secrets files is expected.
